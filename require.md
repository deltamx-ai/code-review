# AI Code Review 需求文档

## 1. 核心目标
- [x] **核心定位**：基于充足的上下文（而非仅代码 Diff）进行 Review，聚焦发现真实的业务逻辑与高危 Bug。
- [x] **边界约束**：严格限制 AI 输出，忽略纯格式、命名风格、无关紧要的重构建议。
- [x] **角色定位**：作为“第一轮筛错、漏洞提示、边界补充”的辅助工具，人类保留最终合并决策权。

## 2. Review 目标层级 (TODO)
系统需支持/覆盖以下 4 个层级的代码审查：
- [x] **基础层**：拦截明显缺陷（代码 Bug、空指针、边界条件、异常处理、死代码）。
- [x] **工程层**：校验项目规范（架构破坏、分层违规、代码风格、性能隐患、可维护性）。
- [x] **业务层**：核对业务逻辑（需求实现偏差、特殊业务规则遗漏、权限/状态流转/计费/风控异常）。
- [x] **风险层**：评估关联影响（模块影响面、兼容性、数据库迁移风险、API 契约变化、联调风险）。

## 3. 上下文获取规范 (TODO)
在发起 Review 前，必须构建并组装以下上下文信息：

### 3.1 必须提供的上下文（P0）
- [x] **变更内容**：完整的代码 Diff（`git diff`）及修改文件列表。
- [x] **业务意图**：一句话说明改动目的（提取自 Issue/PR Title）。
- [x] **业务规则**：当前改动依赖的 3~5 条核心业务规则。

### 3.2 强烈建议提供的上下文（P1）
- [x] **预期行为**：列出程序的正常情况、异常情况及边界情况的表现。
- [x] **项目环境**：技术栈（语言/框架）、数据库、缓存及消息队列等中间件信息。
- [x] **代码上下文**：相关函数的完整上下文代码（避免 AI 产生幻觉）。

### 3.3 按改动类型的特定上下文（动态收集）
- [x] **服务端改动**：补充异常响应结构、事务边界、并发访问情况、上下游依赖影响。
- [x] **数据库改动**：补充表结构、索引变更、数据量级、是否在线迁移、回滚方案。
- [x] **前端改动**：补充页面交互预期、状态管理方式、浏览器兼容要求、埋点/权限/路由规则。
- [x] **基建/脚本改动**：补充执行环境、触发条件、权限范围、失败回退策略。

## 4. 场景落地流程 (TODO)
支持不同量级的 Review 场景：
- [x] **轻量版（日常 PR）**：自动提取 [PR Diff + 改动目的 + 核心业务规则 + 指定重点]，输出高/中/低风险。
- [x] **标准版（团队 CI 流程）**：自动抓取 [Diff + 相关上下文函数 + Issue描述 + 测试结果] -> 套用标准 Prompt -> AI 首轮 Review -> 人类二审高风险问题。
- [x] **高价值版（核心业务）**：在标准版基础上，动态挂载 [历史事故库、类似模块避坑指南、安全/性能红线基线]。

## 5. AI 输出格式约束 (TODO)
Review 结果必须严格按以下结构化格式输出：
- [x] 1. **高风险问题**（优先展示漏洞、业务逻辑错误）
- [x] 2. **中风险问题**
- [x] 3. **低风险优化建议**
- [x] 4. **缺失的测试场景**
- [x] 5. **总结结论**

## 6. 核心防踩坑策略 (TODO)
- [x] **防上下文过载**：禁止 Review 整个大仓库，采用“先审当前 PR -> 再单独深挖高风险文件”的策略。
- [x] **防无效建议**：Prompt 必须包含限制条件，屏蔽“考虑抽离个函数”这类无价值废话。
