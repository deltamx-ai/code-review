# AI Code Review 需求文档 v2

> 目标：在现有 `code-review` CLI 基础上，补齐与 `require.md` 的关键差距，把当前“信息组织 + prompt 驱动”的工具，升级为一个**具备约束、校验、分层能力、结构化输出能力，并可逐步演进到 Web/团队化使用**的系统。

---

## 1. 文档背景

当前项目已经具备以下基础能力：
- 能从 `git diff` 获取变更内容与变更文件
- 能拼装业务目标、业务规则、预期行为、上下文文件
- 能按 `lite / standard / critical` 三种模式生成 Prompt
- 能调用本地 `copilot` CLI 执行 review
- 支持 Jira enrich、上下文扩展、两阶段 `deep-review`
- 支持上下文预算控制、跳过 binary / 非 UTF-8 / 超大文件

但当前系统仍存在以下核心问题：
- 四层审查目标主要体现在 prompt 文案中，尚未形成程序级能力闭环
- P0 / P1 上下文要求没有被强制执行
- 三种场景模式仅实现了命名和部分流程，尚未完整落地
- 输出格式依赖模型自觉遵守，缺少结构化解析与校验
- 当前仅为 CLI，本地单机可用，尚未具备网页访问与团队使用能力

本文件用于定义下一阶段的正式实现目标。

---

## 2. 总体目标

### 2.1 产品目标
将系统从“Prompt 生成工具”升级为“受约束的 AI Code Review 系统”，满足以下要求：

- **输入侧有约束**：关键上下文缺失时，不允许进入正式 review
- **分析侧有分层**：基础层 / 工程层 / 业务层 / 风险层形成明确能力边界与处理流程
- **输出侧有结构**：模型输出必须被解析、校验、归一化，而不是只输出自由文本
- **流程侧可落地**：支持 lite / standard / critical 三种使用场景的真实差异化流程
- **架构侧可演进**：先支持 CLI 增强，再支持 HTTP API，最终支持网页访问

### 2.2 非目标
本阶段不要求：
- 完整替代人工 Code Review
- 覆盖全语言、全框架、全场景的静态分析能力
- 一次性构建完整企业级平台

本阶段重点是：**把核心闭环做实**。

---

## 3. 核心差距与改造方向

### 3.1 四层审查目标缺少程序级闭环
当前问题：
- 基础层、工程层、业务层、风险层主要存在于 prompt 文字要求中
- 系统本身没有显式的分层审查流程和规则挂载机制
- 业务层与风险层几乎完全依赖模型能力，缺少系统兜底

改造目标：
- 系统内部必须显式区分四层审查
- 每层必须定义：输入、检查关注点、补充上下文策略、输出字段
- 对部分高价值风险提供程序级辅助分析，而不是完全依赖 prompt

### 3.2 P0 / P1 上下文校验不够硬
当前问题：
- `validate_args()` 采用打分制，更多是“建议补充”
- P0 项（diff / 业务意图 / 业务规则）并未真正强制
- 缺少“阻断 review”与“降级 review”的规则

改造目标：
- 建立强校验机制
- P0 缺失时禁止进入 `standard` / `critical` 正式 review
- P1 缺失时允许继续，但必须在结果中标记“置信度下降 / 结论受限”

### 3.3 三种模式只有名称，没有完整流程分化
当前问题：
- `lite / standard / critical` 已存在，但能力差异不够清晰
- `standard` 缺少团队流程语义
- `critical` 缺少基线/事故/红线等知识挂载能力

改造目标：
- 三种模式必须具备不同的输入要求、上下文要求、输出约束、执行流程
- `critical` 必须比 `standard` 多出明确的高价值能力，不只是 prompt 更长

### 3.4 输出格式未被程序强制约束
当前问题：
- Prompt 要求 1~5 结构输出，但程序不解析结果
- 缺少结构化 schema、分段检查、字段合法性检查
- 模型输出一旦跑偏，系统无法纠正或二次修复

改造目标：
- 定义统一的 Review Result Schema
- 模型输出后必须进行结构化解析、修复、校验
- 对不符合格式的结果进行二次纠正或报错

### 3.5 当前仅是 CLI，不支持网页访问
当前问题：
- 当前系统只适合本机命令行使用
- 无 API，无任务管理，无 Web UI
- 不支持多用户 / 团队共享 / 浏览器访问

改造目标：
- 保留 CLI 能力
- 在核心逻辑之上新增 HTTP API
- 为后续网页访问提供架构基础

## 5. 四层审查能力设计

### 5.1 基础层（Basic Review Layer）
目标：发现显性缺陷与边界错误。

必须覆盖：
- 空指针 / nil / None / unwrap 风险
- 越界 / 边界条件遗漏
- 异常处理缺失
- 明显死代码 / 不可达逻辑
- 资源未释放 / 错误返回丢失
- 并发下基础竞态风险（如重复写、锁遗漏）

实现要求：
- Prompt 中明确要求检查上述项目
- 结合 diff + context 做文件级审查
- 对常见语言关键词可做轻量规则提示（可选）

输出要求：
- 每条问题必须带定位信息
- 必须标明触发条件与影响

### 5.2 工程层（Engineering Review Layer）
目标：发现工程实现上的结构性问题。

必须覆盖：
- 架构分层破坏
- 模块职责混乱
- 可维护性明显恶化
- 高风险性能隐患
- 配置 / 常量 / 依赖使用不当
- 契约边界不清晰

实现要求：
- 支持项目级规则配置（后续可扩展）
- 支持按文件路径/目录推断层级角色
- 支持检查是否出现跨层直接调用、错误依赖方向等典型模式

最低落地要求：
- 即便暂时没有完整规则引擎，也必须抽象出 `engineering checks` 模块
- 至少支持“目录层级规则 + 关键关键词规则 + 模型审查提示”的组合能力

### 5.3 业务层（Business Review Layer）
目标：检查需求是否真正被正确实现。

必须覆盖：
- 需求实现偏差
- 业务规则遗漏
- 权限校验遗漏
- 状态流转错误
- 幂等性问题
- 计费 / 风控 / 审批类关键逻辑偏差

实现要求：
- 必须依赖业务意图 + 业务规则 + 预期行为
- 若业务规则缺失，不允许进入 `standard` / `critical` 正式 review
- 支持从 Jira / Issue / 外部命令补充 acceptance / rule 信息

最低落地要求：
- 构建 `business context pack`
- review 时显式输出“本次使用了哪些业务规则作为判断依据”

### 5.4 风险层（Risk Review Layer）
目标：评估改动影响面与发布风险。

必须覆盖：
- 模块影响面
- API 契约变化风险
- 数据库迁移风险
- 向后兼容风险
- 联调风险
- 灰度 / 回滚 / 上下游依赖风险

实现要求：
- 支持按改动类型触发不同风险检查模板
- DB 改动必须有迁移风险分析入口
- API / contract 改动必须提示兼容性检查
- critical 模式下必须加载基线/红线信息

最低落地要求：
- 抽象 `risk analyzer`
- 支持基于文件后缀、目录、变更类型推断风险清单

## 6. 上下文校验与准入机制

### 6.1 上下文分级

#### P0：必须提供
- 变更内容：完整 diff + 修改文件列表
- 业务意图：一句话改动目的
- 业务规则：3~5 条核心规则

#### P1：强烈建议提供
- 预期行为：正常 / 异常 / 边界
- 技术栈 / 项目环境
- 相关函数 / 文件上下文
- 测试结果
- Issue / 需求描述

#### P2：按场景增强
- Jira 描述 / 验收条件
- 历史事故库
- 红线 / 基线文件
- 发布说明 / 回滚说明

### 6.2 准入规则

#### Lite 模式
- 必须有 diff
- goal / rules 缺失时允许继续
- 但输出必须标记“低置信度”

#### Standard 模式
- 缺少任一 P0，直接拒绝正式 review
- 若 P1 缺失超过 2 项，允许继续但必须告警

#### Critical 模式
- 必须满足全部 P0
- 必须满足至少 2 项 P1
- 必须提供至少一类 P2 增强信息（baseline / incident / risk focus）
- 不满足时直接阻断

### 6.3 系统行为要求
- `validate` 命令必须返回：
  - 是否可执行 review
  - 缺失项
  - 阻断原因
  - 降级建议
- `run` / `deep-review` / `review` 在正式模式下必须复用相同准入检查
- 不允许存在“validate 说不行，但 review 还能强跑”的情况

## 7. 三种模式的真实落地要求

### 7.1 Lite 模式
适用场景：日常 PR 快速首轮筛查。

输入要求：
- diff 必需
- goal / rules / issue 可选

流程要求：
- 自动提取 diff 与改动文件
- 自动扩展少量上下文
- 输出高 / 中 / 低风险

输出要求：
- 允许简化版结果
- 必须标记输入完整度与结论置信度

### 7.2 Standard 模式
适用场景：团队常规 review / CI 前置审查。

输入要求：
- 全部 P0 必需
- 建议提供 Issue、测试结果、上下文函数

流程要求：
- 自动抓取 diff
- 自动组装上下文
- 自动 enrich Issue / Jira 信息
- 执行首轮 review
- 必要时扩展高风险上下文
- 输出结构化结果
- 标记“建议人工复核的高风险项”

输出要求：
- 必须 machine-readable
- 必须可被 CI 消费

### 7.3 Critical 模式
适用场景：核心业务 / 支付 / 风控 / 数据库迁移 / 权限改动。

输入要求：
- 满足 standard 全部要求
- 额外要求 baseline / incident / redline / risk focus 至少一种

流程要求：
- 在 standard 基础上附加高价值知识挂载
- 强制两阶段 review
- 强制输出风险影响面
- 强制列出发布前需人工确认项

输出要求：
- 必须包含：
  - 高风险问题
  - 风险影响面
  - 缺失验证项
  - 发布建议 / 回滚关注点

## 8. 结构化输出设计

### 8.1 统一输出 Schema
系统内部必须定义统一 Review Result 结构，例如：

```json
{
  "mode": "standard",
  "input_score": 78,
  "input_ok": true,
  "confidence": "medium",
  "high_risk": [
    {
      "title": "重复提交下可能发生双写",
      "file": "src/order/service.rs",
      "location": "create_order",
      "reason": "缺少幂等保护，事务外写入",
      "trigger": "网络重试或并发提交",
      "impact": "重复订单/重复扣款",
      "suggestion": "增加唯一约束或幂等键校验"
    }
  ],
  "medium_risk": [],
  "low_risk": [],
  "missing_tests": [],
  "summary": "发现 1 个高风险问题，建议人工重点复核。",
  "needs_human_review": true,
  "used_rules": ["一个订单只能支付一次", "幂等键必须生效"]
}
```

### 8.2 模型输出处理要求
模型输出后必须经过以下步骤：
1. 原始文本保留
2. 尝试解析为结构化结果
3. 若解析失败，触发一次格式修复
4. 若修复仍失败，则返回错误并提示格式不合规

### 8.3 程序级校验要求
必须检查：
- 是否包含高/中/低风险、测试场景、总结五大部分
- 各问题项是否包含定位、原因、触发条件、影响、建议
- `summary` 不可为空
- critical 模式下必须包含风险影响面与人工确认项

## 9. 知识挂载与增强上下文

### 9.1 Baseline / Redline 支持
系统应支持加载如下外部文件：
- 安全红线
- 性能红线
- 数据库变更规范
- 核心模块避坑指南

要求：
- `critical` 模式可通过参数显式挂载
- 后续可通过配置自动挂载

### 9.2 Incident / 事故库支持
系统应支持挂载历史事故摘要，用于 critical 模式增强审查。

最低要求：
- 支持通过文件输入事故摘要
- 支持在 prompt 中单独注明“历史事故参考”

### 9.3 Change-Type 特定增强
按改动类型动态挂载额外检查项：
- server
- db
- frontend
- infra
- contract/api（新增建议）

## 10. CLI 改造要求

### 10.1 命令行为一致性
以下命令必须统一使用同一套校验与输出机制：
- `validate`
- `run`
- `deep-review`
- `review`

### 10.2 新增建议命令
建议新增：
- `review-json`：输出结构化 JSON review 结果
- `review-raw`：输出原始模型文本
- `admission-check`：只做准入校验
- `analyze-change-type`：分析当前改动类型与风险模板

### 10.3 输出格式
所有核心命令应支持：
- `text`
- `json`

其中 JSON 输出必须稳定、可供脚本消费。

## 11. HTTP API / Web 化改造要求

> 本部分为下一阶段能力，但本次设计必须提前兼容。

### 11.1 服务化目标
在不破坏 CLI 的前提下，将核心能力抽象为 service 层，以支持：
- CLI 调用
- HTTP API 调用
- 后续 Web UI 调用

### 11.2 必备 API
建议提供：
- `POST /api/validate`
- `POST /api/prompt`
- `POST /api/run`
- `POST /api/deep-review`
- `GET /api/models`
- `GET /api/auth/status`

### 11.3 任务模型
对于耗时任务，建议采用异步任务模型：
- 提交任务返回 `task_id`
- 前端轮询结果
- 保留任务状态：pending / running / done / failed

### 11.4 安全要求
Web 化后必须考虑：
- repo 路径白名单
- 文件读取边界
- token / session 管理
- 多用户隔离
- 审查记录隔离

## 13. 验收标准

### 13.1 输入准入验收
- [ ] standard 模式缺少 diff 时必须拒绝执行
- [ ] standard 模式缺少 goal 时必须拒绝执行
- [ ] standard 模式缺少 rules 时必须拒绝执行
- [ ] critical 模式缺少 baseline/incident/risk focus 时必须拒绝执行
- [ ] lite 模式上下文不足时允许继续，但结果中必须标记低置信度

### 13.2 输出结构验收
- [ ] review 结果必须可输出为 JSON
- [ ] JSON 中必须包含 high_risk / medium_risk / low_risk / missing_tests / summary
- [ ] 每个高风险问题必须包含 file/location/reason/trigger/impact/suggestion
- [ ] 输出不合规时系统必须报错或自动修复，而不是直接放过

### 13.3 模式行为验收
- [ ] lite / standard / critical 在准入规则上必须有显式差异
- [ ] critical 模式必须强制两阶段 review
- [ ] critical 模式必须输出影响面与人工确认项

### 13.4 架构演进验收
- [ ] CLI 核心逻辑可被 service 层复用
- [ ] 后续可无痛接入 HTTP API

## 14. 建议的代码改造方向

建议新增/重构模块：
- `src/admission.rs`：准入校验与阻断逻辑
- `src/review_schema.rs`：结构化输出 schema
- `src/review_parser.rs`：模型输出解析与修复

- `src/risk.rs`：风险分析器
- `src/services/`：供 CLI / API 共用的 service 层
- `src/web/`：后续 HTTP API

## 15. 一句话总结

下一阶段不是继续“优化 prompt 文案”，而是要把系统补成一个**有准入、有分层、有结构化结果、有模式差异、有演进空间**的真实 AI Code Review 产品内核。
